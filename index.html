البنية التحتية للهاردوير

يعتمد الروبوت على متحكم دقيق ESP32 كعقل مفكر، متصل بمشغلات مواتير عالية الطاقة BTS7960 لتحمل تيارات المواتير، وحساس أشعة تحت الحمراء خماسي (5-Channel IR) لاستشعار الخط بدقة عالية.

متحكم ESP32

وحدة المعالجة المركزية، تدير إشارات الـ PWM، قراءة الحساسات، والاتصال بالبلوتوث.

درايفر المواتير BTS7960

يوفر طاقة عالية (حتى 43 أمبير). نستخدم اثنين، واحد للجانب الأيمن وآخر للأيسر.

حساس الخط 5-Channel

خمسة حساسات رقمية تغطي مساحة واسعة لتوفير بيانات دقيقة لخوارزمية التحكم.

دورة حياة الروبوت (FSM)

1

انتظار الأمر (WAITING_FOR_1)

الروبوت متوقف تماماً، ينتظر إرسال رقم '1' عبر البلوتوث.

2

تتبع الخط - المرحلة الأولى

يبدأ الحركة باتباع الخط باستخدام خوارزمية PD حتى تعثر الحساسات الخمسة على علامة التقاطع.

3

توقف مؤقت وإرسال إشارة

يتوقف لمدة 3 ثوانٍ، يرسل '2' للهاتف، ويدخل في وضع انتظار الأمر '3'.

4

تتبع الخط - المرحلة الثانية

عند استلام '3'، يكمل التتبع حتى يرى التقاطع مرة أخرى.

5

الإنهاء (العودة للبداية)

يتوقف 3 ثوانٍ، يرسل '0' للهاتف، ويعود للحالة الأولى.

خوارزمية التحكم (PD Control)

لجعل الروبوت يتحرك بنعومة دون اهتزازات (Zig-Zag)، نستخدم معادلة رياضية تعتمد على الخطأ الحالي والخطأ السابق:

Correction = (KP * Error) + (KD * (Error - lastError))
  • KP (0.30): يحدد قوة الدوران للعودة للخط.
  • KD (0.08): يمتص الصدمات ويمنع الروبوت من تخطي الخط.

معالجة التوقف (Stall Prevention)

محركات DC الثقيلة تصدر طنيناً وتتوقف إذا كانت إشارة الـ PWM ضعيفة جداً. تم تطبيق كود برمجي لمنع السرعة من الانخفاض عن `MIN_EFFECTIVE_PWM = 100` لضمان الحركة الدائمة.

توثيق المشكلة: عطل درايفر الموتور

أثناء التجارب، لاحظنا أن المحرك الأيسر يعمل بشكل ممتاز عند إعطاء أمر الرجوع للخلف، ولكنه يتوقف تماماً عند إعطاء أمر التقدم للأمام، بالرغم من سلامة الكود.

عمل الموتور في اتجاه واحد يثبت أن الموتور نفسه سليم والبطارية توفر طاقة كافية. المشكلة انحصرت في مسار الإشارة (Forward Path) داخل دائرة جسر-H (H-Bridge) في شريحة BTS7960 المسؤولة عن الجانب الأيسر، أو في السلك الموصل بـ GPIO 27.

بما أن الروبوت مخصص لتتبع الخطوط (Line Follower) ولا يحتاج للرجوع للخلف أبداً، قمنا بخدعة هندسية:

تم عكس الوظائف برمجياً: استخدمنا الأقطاب السليمة التي كانت مخصصة للرجوع للخلف (Pins 14 & 26) وجعلناها هي المسؤولة عن التقدم للأمام، وتم تجاهل الأقطاب التالفة تماماً دون الحاجة لشراء درايفر جديد!

المشكلة: الروبوت كان يصدر صوتاً (أزيز) ولا يتحرك عند السرعات 1، 2، 3 عبر البلوتوث، وبدأ يتحرك بانتظام عند السرعة 4.
التفسير: الدرايفر والمحركات تتطلب طاقة أولية للتغلب على القصور الذاتي والوزن. السرعة 4 تعادل إشارة PWM بقيمة (113).
الحل: تم ضبط المتغير الأساسي baseStraight = 115 وتم رفع حد الأمان البرمجي لمنع نزول الإشارة عن 100 لتجنب توقف المحرك المفاجئ.

ESP32_LineFollower

أوامر الوضع الآلي (Auto Mode)

  • 1 بدء التتبع (المرحلة الأولى)
  • 3 استكمال التتبع (المرحلة الثانية)
  • S توقف طوارئ (Emergency Stop)

التبديل بين الأوضاع

  • n تفعيل التحكم اليدوي (Manual Override)
  • N العودة للوضع الآلي وانتظار '1'

كود التبديل بين الأوضاع 

void handleBluetooth() {
  if (!SerialBT.available()) return;
  char c = SerialBT.read();

  // التبديل للوضع اليدوي
  if (c == 'n') {
    currentState = MANUAL_CONTROL;
    stopRobot();
  } 
  // العودة للوضع الآلي
  else if (c == 'N') {
    currentState = WAITING_FOR_1;
    stopRobot();
  }

  // .. باقي الأوامر بناءً على الحالة الحالية
}
ميزة هامة: فصلنا منطق الـ FSM إلى نظامين متوازيين لمنع تعارض الأوامر. إذا كان الروبوت في الوضع اليدوي MANUAL_CONTROL، فإنه يتجاهل أمر '1' تماماً حتى ترسل له 'N' ليعود لبرنامجه الآلي.
